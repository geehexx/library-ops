# Library Ops Core Spec

## Status

Supporting Spec Kit artifact. The canonical product contract is `.taskmaster/docs/prd.md`; this file summarizes the implementation slice for Spec Kit consistency and agent execution.

## User-visible goal

Build a polished Django library operations demo where an evaluator can log in as Admin, Librarian, or Member; manage catalog records; borrow and return copies; search by exact identifiers, keywords, subjects, and semantic phrases; and see clear provenance for seeded/AI-assisted metadata.

## Primary participants

- **Admin**: manages users, roles, catalog records, and system evidence.
- **Librarian**: manages catalog/inventory and performs circulation workflows.
- **Member**: searches catalog, views availability, and sees their own loans.
- **Evaluator**: verifies assignment coverage, local setup, deployed demo, and quality evidence.

## Core scenarios

1. Admin/librarian creates or edits a bibliographic work, edition, contributors, subjects, and copy records.
2. Member searches by ISBN/barcode/title/author/subject/natural-language phrase and sees availability.
3. Librarian borrows an available copy for a member.
4. Librarian returns an active loan.
5. Member cannot mutate catalog or circulation by UI, direct POST, or API call.
6. Seed commands rebuild demo data deterministically with provenance.
7. AI metadata suggestions are reviewed before persistence and marked with provenance.

## Non-negotiable requirements

- Work, edition, copy, and loan are separate concepts.
- A copy cannot have more than one active loan.
- Search exact identifiers outrank lexical/vector matches.
- Availability is derived from database state, not a hand-edited display flag.
- Authorization is enforced server-side.
- Seed data is generated by idempotent management commands, not committed corpora.
- The selected Codex/Task Master/Spec Kit/RTK/code-intelligence toolchain is required for implementation environments.
